# Отчёт — финальный проект (Трек B)

**Тема:** двухуровневый Q&A-ассистент по статьям Habr про RAG: Local RAG + Stack Overflow fallback.

**Запуск:** `python eval.py`

## 1. Задача и данные

Задача — построить CLI-систему, которая отвечает на вопросы по корпусу статей про RAG. Базовый путь ответа идёт через локальный RAG по Habr. Если вопрос просит практический инженерный совет или сравнение с внешней практикой, агент подключает второй инструмент — Stack Overflow fallback.

Данные лежат в `input/`:

| Параметр | Значение |
|---|---:|
| Habr-документов | 10 |
| Чанков BM25 | 152 |
| Символов в чанках | 153 464 |
| Stack Overflow fallback | live Stack Exchange API, cache только при ошибке/офлайн |
| Gold-набор | 20 вопросов |

Корпус Habr включает статьи про Qwen3-Embedding, OCC-RAG, DCD, внешнюю память LLM, RAGAS, LLM monitoring и локальную инфраструктуру. Stack Overflow сделан live-ручкой через Stack Exchange API: по умолчанию `stackoverflow_tool.py` ходит в сеть, получает вопросы и accepted/top answers, а `input/stackoverflow_cache.json` используется только при ошибке API или если явно выставить `STACKOVERFLOW_MODE=cache`.

Gold-набор:

- 5 `local_only`: ответ должен быть в Habr-корпусе;
- 5 `stackoverflow`: практический совет берётся из Stack Overflow API;
- 5 `mixed`: нужен multi-hop через Habr + Stack Overflow;
- 5 `out_of_scope`: логичные вопросы по области, но ответа нет в источниках. Они проверяют, умеет ли система отказаться.

## 2. Система

Техники курса:

| # | Техника | Где | Как работает |
|---|---|---|---|
| 1 | RAG | `rag.py` | BM25-индекс по Habr-статьям, top-k чанки с источниками |
| 2 | Агент с инструментами | `agent.py` | planner выбирает `local_rag`, `stackoverflow_search` или оба |
| 3 | PWC / мультиагентный паттерн | `agent.py` | planner → parallel workers → critic |
| 4 | Структурированный вывод | `schema.py` | `AssistantAnswer`, `Evidence`, `JudgeDecision`, `EvalRow` |
| 5 | LLM-as-judge | `judge.py` | при `OPENAI_API_KEY` запускается LLM-судья; без ключа — deterministic fallback |
| 6 | Проверка галлюцинаций | `agent.py`, `eval.py` | ghost quotes и unsupported numbers считаются отдельно |

`field_validator` и бизнес-инвариант находятся в `schema.py`: `status="answered"` требует `confidence >= 0.45` и evidence, а `status="insufficient"` требует `confidence <= 0.35`.

Схема пути данных:

1. `eval.py` читает вопрос из `input/gold.json`.
2. Planner сам выбирает маршрут; eval больше не передаёт `forced_type`.
3. Workers вызывают `local_rag` и/или `stackoverflow_search`; mixed-вопросы идут параллельно. Если первый `local_rag` выглядит слабым (top BM25 score ниже 15), агент делает второй локальный RAG-вызов с уточнённым запросом.
4. Synthesizer собирает `AssistantAnswer`.
5. Critic проверяет ghost-цитаты, выдуманные числа и ожидаемый маршрут.
6. `judge.py` выставляет verdict: `supported`, `partially_supported`, `unsupported`, `abstain_ok`.
7. Артефакты пишутся в `output/`.

Из RAG убраны ручные source boosts под конкретные статьи. Поиск ранжирует чанки по BM25 с заголовком документа в индексируемом тексте.

## 3. Оценка

Команда: `python eval.py`.

Критерий pass:

- для in-scope: правильный route, достаточно keyword/source hits, `judge ∈ {supported, partially_supported}`, `ghost_quotes=0`, `unsupported_numbers=0`;
- для out-of-scope: система должна вернуть `status=insufficient` и `judge=abstain_ok`.

Фактический результат live-прогона через Stack Exchange API:

| Тип | Passed | Total |
|---|---:|---:|
| local_only | 4 | 5 |
| stackoverflow | 4 | 5 |
| mixed | 3 | 5 |
| out_of_scope | 0 | 5 |
| **Итого** | **11** | **20** |

После снятия ручных source boosts, запрета `forced_type` в eval и отключения cache-on-empty для Stack Overflow результат стал заметно строже. После отдельной проверки live Stack Exchange API вопросы `stackoverflow` были переписаны под реальные Stack Overflow/RAG-кейсы: Haystack/Milvus pipeline, Graph RAG metrics, RecursiveCharacterTextSplitter, RAG citations и table-based hybrid retrieval. Система стала нормально использовать внешнюю ручку, но не была дожата до 100%: остались ошибки маршрутизации, неполные mixed-ответы и слабый отказ на out-of-scope.

Путь выполнения тоже измеряется, а не описывается словами. В `eval_results.json` и `eval_table.csv/md` для каждого кейса сохранены `actual_route`, `steps`, `search_calls`, `token_estimate`, `latency_sec` и режим Stack Overflow. В финальном прогоне фактическое распределение маршрутов такое:

| Фактический маршрут | Кейсов |
|---|---:|
| `local_rag` | 10 |
| `stackoverflow_search` | 6 |
| `local_rag + stackoverflow_search` | 4 |

Нули в `ghost_quotes` и `unsupported_numbers` объясняются тем, что основной режим извлекательный: цитаты берутся из retrieved context, а не генерируются свободно. Чтобы проверить, что gate не выключен, я добавил synthetic tests:

| Поддельная ghost-цитата | caught |

## 4. Где пайплайн ошибался

### Провал 1 — q07: RecursiveCharacterTextSplitter ушёл в local RAG

**Вход:** «Как настроить RecursiveCharacterTextSplitter: что означают chunk_size и chunk_overlap?»

**Что произошло:** вопрос должен был идти во внешний Stack Overflow tool, но простой planner не распознал `chunk_size` и `chunk_overlap` с подчёркиваниями. Он отправил вопрос в локальный Habr RAG.

**Искажение:** вместо Stack Overflow примеров по `RecursiveCharacterTextSplitter` ответ притянул фрагменты Habr про eval-форматы и внешнюю память. Judge поставил `supported`, потому что цитаты настоящие, но route и содержание не соответствуют вопросу.

### Провал 2 — q14: mixed-вопрос стал только Stack Overflow

**Вход:** «Почему статья про Qwen3-Embedding и настройка RecursiveCharacterTextSplitter говорят, что chunk_size и chunk_overlap надо проверять на eval?»

**Что произошло:** вопрос требует Habr + Stack Overflow, но planner увидел технические маркеры `chunk_size/chunk_overlap` и выбрал только `stackoverflow_search`.

**Искажение:** ответ объяснил только сторону Stack Overflow про `RecursiveCharacterTextSplitter`, но потерял Habr-контекст про Qwen3-Embedding, длинный контекст 32k и влияние модели эмбеддингов на стратегию chunking.

### Провал 3 — q18: цена Claude Sonnet

**Вход:** «Сколько стоит миллион токенов у Claude Sonnet сегодня и как это влияет на стоимость RAG?»

**Что произошло:** вопрос требует актуального внешнего прайса, которого нет ни в Habr-корпусе, ни в Stack Overflow результатах. Система всё равно вернула `status=answered`.

**Искажение:** локальный RAG притянул фразу «А сколько это стоит?» из статьи про домашний сервер и числа про токены Claude Code. Judge поставил `unsupported`. Правильное поведение — `insufficient`.

## 5. Выводы

Двухуровневый маршрут полезен: Habr даёт предметный контекст, Stack Overflow API добавляет практические советы про chunking, hybrid search и проверку цитат. Однако live API и простой BM25 делают результат шумным.

Главная граница — отказ от ответа и multi-hop retrieval. На 4 из 5 out-of-scope вопросов ассистент не сказал «данных недостаточно», а попытался собрать ответ из ближайших фрагментов. В mixed-вопросах часто найден правильный источник, но ответ неполный. В продакшене я бы добавил threshold по retrieval score, reranker, отдельный abstain-policy worker и более строгий LLM-as-judge до отдачи ответа пользователю.

Что оставил бы в прод:

- трассировку route/steps/tools;
- structured output с validator;
- ghost-quote и unsupported-number gates;
- Stack Overflow live API с cache fallback для воспроизводимого eval.

Что переделал бы:

- добавить dense retrieval + reranker вместо одного BM25;
- научить planner выбирать `insufficient` до генерации;
- запускать LLM-as-judge на out-of-scope и low-confidence кейсах;
- сохранять live Stack Overflow результаты в cache с датой и API query.
